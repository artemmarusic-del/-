# ХЕ.Дневник — приложение для диабетиков

Веб-приложение для людей с диабетом: дневник питания с расчётом БЖУ и хлебных единиц (ХЕ),
расчёт дозы инсулина (болюс на еду + коррекция с учётом «активного» инсулина), и модуль
самонастройки, который со временем подстраивает индивидуальные коэффициенты под конкретного
человека на основе истории измерений глюкозы.

> ⚠️ **Важно.** Это инструмент для учёта и ориентировочных расчётов, а не медицинское изделие
> и не замена консультации врача. Любые изменения схемы инсулинотерапии должны обсуждаться
> с лечащим эндокринологом.

## Стек

- **Backend:** Node.js, Express, TypeScript, Prisma ORM, PostgreSQL, JWT-аутентификация
  (httpOnly-cookie), bcrypt, zod-валидация.
- **Frontend:** React + TypeScript, Vite, Tailwind CSS, React Router, Recharts, Zustand.

## Возможности

- Регистрация и вход (email + пароль), данные каждого пользователя изолированы в БД.
- Дневник питания: приёмы пищи из базы продуктов (БЖУ на 100 г) или свои продукты вручную.
- Автоматический расчёт ХЕ и БЖУ приёма пищи.
- Расчёт дозы инсулина: углеводный коэффициент и корректирующий фактor — отдельно для
  утра/дня/вечера/ночи, с учётом «инсулина на борту» (IOB), чтобы не наложить дозы.
- Дневник глюкозы и дневник инсулина, сводка по дню, графики и статистика (время в целевом
  диапазоне, средняя глюкоза, ХЕ по дням).
- **Модуль самонастройки:** анализирует последние ~14 дней дневника по каждому времени суток
  отдельно и предлагает небольшую (не более ±15%) корректировку коэффициентов, если видит
  устойчивое отклонение глюкозы от целевого диапазона. По умолчанию предложения нужно подтвердить
  вручную в разделе «Настройки»; есть переключатель для автоматического применения. Каждое
  изменение сохраняется в истории с объяснением, почему оно было предложено.

## Быстрый запуск (Windows)

Если проект уже настроен (установлены Node.js, PostgreSQL и зависимости) — просто запустите
**ХЕ.Дневник.exe** в корне проекта. Он тихо поднимет оба сервера в фоне и откроет приложение
в отдельном окне без браузерного интерфейса (режим app-окна Chrome/Edge) — выглядит и ведёт
себя как обычная программа. Закрытие окна корректно останавливает серверы.

Исходник лаунчера — `tools/launcher.cs`; пересобрать его можно встроенным в Windows компилятором:

```bat
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\csc.exe /optimize /target:winexe /r:System.Windows.Forms.dll /out:ХЕ.Дневник.exe tools\launcher.cs
```

Запасной вариант — `ЗАПУСК.bat` (то же самое, но окна серверов видны).

## Запуск локально

### 1. Предварительные требования

- [Node.js](https://nodejs.org/) версии 20 LTS или новее (включает npm).
- PostgreSQL — либо через Docker (`docker compose up -d`, конфиг уже в `docker-compose.yml`),
  либо локально установленный сервер.

### 2. Backend

```bash
cd server
cp .env.example .env
```

Откройте `server/.env` и укажите:
- `DATABASE_URL` — строка подключения к вашей PostgreSQL (значение по умолчанию подходит для
  `docker compose up -d` из корня проекта).
- `JWT_SECRET` — длинная случайная строка. Сгенерировать можно командой:
  ```bash
  node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"
  ```

Далее:

```bash
npm install
npx prisma migrate dev --name init   # создаст таблицы в БД
npm run seed                          # заполнит базу продуктов
npm run dev                           # запустит API на http://localhost:4000
```

### 3. Frontend

В отдельном терминале:

```bash
cd client
cp .env.example .env
npm install
npm run dev
```

Приложение откроется на http://localhost:5173 и будет обращаться к API на порту 4000
(при необходимости поменяйте `VITE_API_URL` в `client/.env`).

При первом входе после регистрации приложение попросит заполнить начальный профиль
(вес, целевой диапазон глюкозы, стартовые коэффициенты) — далее их можно уточнить в разделе
«Настройки», а система будет предлагать корректировки сама по мере накопления дневника.

## Деплой в облако

Самый простой вариант — [Render.com](https://render.com) (есть бесплатный уровень):

1. **База данных:** создайте PostgreSQL instance в Render, скопируйте `Internal Database URL`.
2. **Backend:** новый Web Service из папки `server` (или из Docker-образа — `server/Dockerfile`
   уже готов). Переменные окружения: `DATABASE_URL` (из шага 1), `JWT_SECRET`, `CLIENT_URL`
   (адрес будущего фронтенда), `NODE_ENV=production`. Build command: `npm install && npm run build`.
   Start command: `npx prisma migrate deploy && npm start`.
3. **Frontend:** новый Static Site из папки `client`. Build command: `npm install && npm run build`,
   папка публикации: `dist`. Переменная окружения `VITE_API_URL` — адрес backend-сервиса + `/api`.
4. После деплоя один раз выполните `npm run seed` для backend-сервиса (через Render Shell),
   чтобы заполнить базу продуктов.

Аналогично можно развернуть на Railway, Fly.io или любом хостинге с поддержкой Node.js и
PostgreSQL — принцип тот же: три независимых сервиса (БД, API, статический фронтенд).

```

```

## Как устроен расчёт дозы инсулина

```
ХЕ = граммы_углеводов / граммы_на_1_ХЕ   (по умолчанию 10 г, настраивается)
Доза на еду = ХЕ × (единиц инсулина на 1 ХЕ, для текущего времени суток)
Коррекция   = (текущая_глюкоза − целевой_уровень) / фактор_коррекции   — только если глюкоза выше целевого диапазона
Итоговая доза = Доза на еду + max(0, Коррекция − уже_активный_инсулин)
```

«Активный инсулин» считается по линейному затуханию от предыдущих доз в пределах заданной
длительности действия инсулина (по умолчанию 4 часа), чтобы не советовать колоть повторно поверх
ещё не отработавшей дозы.

## Как работает самонастройка

Раз в день (или по кнопке «Проанализировать дневник сейчас» в Настройках) приложение отдельно
для каждого времени суток:

- Сравнивает глюкозу через 1.5–3 часа после еды с целевым диапазоном — если она стабильно выше
  или ниже нормы (минимум 5 наблюдений за 14 дней), предлагает увеличить/уменьшить углеводный
  коэффициент на 15%.
- Сравнивает фактическое снижение глюкозы после корректирующих доз (без еды) с ожидаемым —
  если инсулин действует слабее или сильнее, чем предполагает текущий фактор коррекции,
  предлагает его скорректировать.
- Не предлагает менять один и тот же коэффициент чаще, чем раз в 5 дней, и требует подтверждения
  пользователем, если не включён режим автоматического применения.
